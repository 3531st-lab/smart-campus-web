# 课表定位与运行时加固规格

## Problem

2025-2026 学年第二学期的旧版 Excel、OCR 或历史缓存数据可能把节次保存为“第1-2节”“1-2”“2节”等文本。当前前端和服务端分别直接调用 `Number(...)`，文本会变成 `NaN`。生成无效的 CSS `grid-row` 后，浏览器使用网格自动布局，把课程卡片放到显式 12 节网格之后，看起来像整张课表下沉到底部。

课表解析逻辑同时散落在前端归一化、前端渲染、时间标签、OCR 和 Excel 导入中，修改一处不能覆盖全部入口，增加了回归风险和编码成本。项目此前也没有课表定位的自动化测试。

## Solution

新增一个浏览器与 Node.js 共用的课表节次核心模块，集中完成文本解析、范围推断、边界约束和 CSS 网格定位。前端归一化与渲染、服务端 Excel/OCR 导入都调用同一实现。为数字值、中文节次、范围、OCR 文本和非法输入建立快速 Node 测试，并将测试加入默认检查命令。

同时进行定向安全与性能审计：检查生产签名密钥、跨域策略、危险 DOM 注入点、依赖漏洞、静态资源体积和课表接口延迟；只在有明确证据且不破坏现有部署的情况下修改运行时配置。

## User Stories

- 作为学生，我导入旧版中文课表后，课程应显示在正确节次，而不是沉到课表底部。
- 作为学生，我切换 2025-2026 学年第二学期的任意周次时，课程定位应保持一致。
- 作为管理员，我通过 OCR 或 Excel 导入“第1-2节”格式时，不需要手工改成数字。
- 作为维护者，我修改节次解析规则时，可以运行秒级测试确认所有入口保持兼容。
- 作为运维人员，生产环境不得在缺少签名密钥时静默使用公开默认密钥。

## Implementation Decisions

- 使用 UMD 风格的 `public/timetable-core-v155.js`，浏览器通过 `window.TimetableCore` 使用，Node.js 服务端和测试通过 `require` 使用，避免复制解析逻辑。
- 页面入口统一加载规范文件 `app.js?v=155`；旧版本文件仅暂留兼容，不再作为后续开发入口。
- 支持数字、`第 N 节`、`N-M`、`第 N-M 节`、`N 至 M` 和带“节”后缀的连续节数。
- 开始节次限制为 1-12，连续节数限制为 1-4，并保证课程不会超出第 12 节。
- 渲染层不再自行做数字转换，只消费核心模块生成的有限整数网格位置。
- 保留现有 UI、课程配色、学期筛选和数据结构，降低修复影响面。
- 本次不拆分完整 35 万字节前端单文件；先抽出高风险课表领域逻辑，后续再按页面渐进模块化。

## Testing

- 单元测试：数字节次保持不变。
- 单元测试：`第1-2节` 与 `2节` 输出第 2 行开始、跨 2 行。
- 单元测试：`1-3` 自动推断连续 3 节。
- 单元测试：第 11 节连续 4 节被约束到第 12 节以内。
- 单元测试：非法文本始终输出有限整数，不触发 CSS 网格自动安置。
- 静态检查：服务端与共享模块均通过 `node --check`。
- 浏览器回归：本地课表页面无 `NaN` 网格样式，课程卡片行号均在 2-13 之间。
- 性能基线：记录关键静态资源压缩体积和本地课表接口多次请求延迟。

## Out of Scope

- 重写整个前端为 React/TypeScript。
- 改变现有数据库表结构或课程业务规则。
- 自动纠正无法从图片可靠识别的课程名称、教师和教室。
- 本次直接替换生产数据库或第三方部署平台。
