# 全站回归检查与等保预评估规范

## Problem

网站同时运行在本地 Node、Vercel API 与 Cloudflare Pages 三条链路上，且包含身份库、课表、实验室预约、消息、AI 与支付等敏感功能。当前缺少覆盖部署边界的统一测试，线上错误曾直接显示数据库信息，缓存策略也可能让登录用户看到过期数据。课表历史数据还出现过无效节次导致课程卡片下沉到底部的问题。

## Solution

建立可重复的四层检查：静态语法与依赖审计、HTTP/API 冒烟、浏览器关键流程、依据 GB/T 22239-2019 与 GB/T 28448-2019 的非破坏性等保预评估。对已证实的问题先增加回归测试，再修复错误脱敏、请求边界、缓存、限流状态和课表定位；输出诊断记录、风险分级和后续整改清单。

## User Stories

- 作为学生，我登录后应稳定看到自己的课表、消息和预约，不受旧缓存或其他账号数据影响。
- 作为管理员，我应能进入身份库并执行筛选、分页和导入，而游客必须被服务端拒绝。
- 作为运维人员，我需要 5xx 错误不泄露数据库账号、路径、密钥或上游内部信息。
- 作为维护者，我需要一次命令完成语法、单元、安全与核心 API 回归检查。
- 作为安全负责人，我需要一份能映射等保控制域、证据和整改优先级的预评估报告。

## Implementation Decisions

- 正式等保测评必须由有资质的测评机构实施；本次只做非破坏性技术预评估，不执行压力攻击、暴力破解或生产数据篡改。
- 使用 Node 内置测试与临时随机端口，不依赖外部数据库和第三方服务，保证诊断结果可重复。
- 生产环境的未知 5xx 一律返回通用消息；安全日志保留内部错误并附带请求上下文。
- 登录态 API 默认 `no-store`，避免浏览器缓存个人信息；公开静态资源继续使用内容缓存。
- 限流窗口使用有容量上限的注册表，清理过期键，防止长进程内存持续增长。
- 课表定位继续由共享解析核心负责，前端、服务端导入与测试只消费同一结果。

## Testing

- 运行 `npm test`，覆盖课表节次、错误脱敏、限流容量、边缘代理策略和游客核心流程。
- 运行 `npm run check` 与 `npm audit --omit=dev`。
- 检查 HTML/CSS/JS 资源、CSP、安全响应头、CORS、错误返回和缓存头。
- 对首页、登录、身份库、课表、实验室预约、消息、校园资讯、软件库、学习工具与 AI 页面做浏览器冒烟。
- 记录本地接口状态、响应时间和资源体积；部署环境只做低频、非破坏性验证。

## Out of Scope

- 不冒充有资质机构出具正式等保测评结论或备案证明。
- 不在生产环境执行拒绝服务、口令爆破、SQL 注入破坏或大规模爬取。
- 不在本轮整体重写为 React/TypeScript，也不替换现有数据库与部署平台。
- 不自动迁移现有用户令牌到新认证系统；会记录 HttpOnly Cookie 迁移建议。

## Further Notes

完成标准是：所有新增回归测试通过；已确认的高风险信息泄露与缓存错误得到修复；课表下沉测试保持通过；等保预评估报告明确已满足、部分满足、未验证和需第三方验证的项目。
