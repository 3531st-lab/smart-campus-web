# 班级分组与校园群聊设计

## 背景与目标

当前学生身份库已经保存学校、学院、专业和班级，但账号列表按更新时间排序，同班成员会被分页和更新顺序打散。校园生活也缺少可持久化、可治理的群聊能力。

本次目标：

- 学生身份库以班级为主要顺序，同班账号连续展示。
- 使用“学校 + 学院 + 班级”唯一确定班级，专业不参与唯一性判断。
- 每个有效班级必须存在一个系统班级群，学生按身份信息自动归群。
- 普通用户可以创建普通群，并通过邀请、群号申请或随机二维码申请加入。
- 班长、团支书、班级管理员和班主任管理本班群聊；平台管理员可隐身治理所有群聊。
- 群聊支持可靠消息、未读状态、断线补偿、冻结申诉和可扩展的实时传输。
- 提供基础表情、用户图片表情和合规网络表情收藏能力。

## 不在首版范围

- 不复制 QQ、微信的专有表情素材、视觉资源或协议实现。
- 不开放匿名群、游客群聊或未登录用户搜索成员。
- 不支持语音通话、视频通话、直播、红包和端到端加密。
- 不把手机号或其他敏感身份信息暴露给群成员。
- 不允许班级群被普通成员退出、转让或解散。

## 方案选择

采用 MySQL 持久化、REST 可靠操作和可切换实时传输的混合方案：

1. MySQL 是群、成员、申请、消息、已读位置、申诉和审计记录的唯一数据源。
2. REST API 承担所有写操作和历史查询，写入数据库成功后才能确认发送成功。
3. Cloudflare 实时网关按群建立 WebSocket 频道，服务端使用内部签名事件通知网关广播。
4. 本地开发、实时网关不可用或 WebSocket 断开时，客户端自动使用增量轮询。
5. 客户端按最后消息序号补取遗漏内容，WebSocket 和轮询共享同一消息模型。

相比直接依赖长连接，该方案兼容现有 Vercel/Cloudflare Serverless 部署；相比纯轮询，它保留真实实时体验并具备可靠降级路径。

## 领域与身份规则

### 班级唯一性

班级由规范化后的学校、学院和班级名称唯一确定。专业仍保存在身份资料中，用于展示、搜索和统计，但不参与班级键计算。

下列记录属于同一个班级：

- 学校相同。
- 学院相同。
- 班级名称相同。
- 即使专业名称不同，也不拆分班级群。

任一字段为空时不自动建班级群，身份库向管理员显示“班级资料不完整”，待修正后再同步。

### 平台角色、班级职务与群角色

三套权限必须独立：

- 平台角色：学生、老师、管理员、总管理员。
- 班级职务：普通成员、班长、团支书、班级管理员、班主任、任课老师。
- 群角色：群主、群管理员、群成员。

设置班长或团支书不会把学生提升为平台管理员。设置老师为班主任、任课老师或班级管理员，只影响对应班级与班级群。

### 学生与老师归班

- 状态正常的学生根据身份库中的学校、学院和班级自动建立班级分配并加入班级群。
- 学生班级变化时，以事务移除旧班级群成员关系并加入新班级群。
- 停用学生不得访问群聊；重新启用后按当前班级恢复成员关系。
- 老师不会仅凭身份库中的班级字段自动入群。
- 管理员必须明确把老师分配为班主任、任课老师或班级管理员，老师才加入对应班级群。
- 一位老师可以被分配到多个班级。

## 学生身份库设计

### 排序和分组

身份库默认按以下顺序查询：

1. 学校。
2. 学院。
3. 班级。
4. 平台角色排序：学生、老师、管理员、总管理员。
5. 班级职务优先级。
6. 姓名和学号/工号。

同班记录在全局排序中连续。页面为每个班级显示分组标题、学生人数、已分配老师人数和资料完整状态。分页仍按稳定排序进行；跨页的同一班级在下一页显示“接上页”标识。

### 管理操作

管理员可以：

- 筛选学校、学院、班级和平台角色。
- 设置学生为班长、团支书或班级管理员。
- 将老师分配到一个或多个班级，并设置班主任、任课老师或班级管理员。
- 批量导入班级职务和老师班级分配。
- 查看班级群同步状态并重试失败任务。

导入模板增加“班级职务”和“关联班级”列。学生的关联班级默认取“班级”列；老师必须填写“关联班级”并提供学校和学院。

## 群聊类型与成员规则

### 班级群

- 每个完整、有效的班级自动拥有且仅拥有一个班级群。
- 群名默认使用“班级名称 + 班级群”，允许班级管理员修改显示名称，但班级标识不变。
- 班级群显示认证标识，固定排在普通群之前。
- 学生不能退出或解散班级群。
- 班长、团支书、班级管理员和班主任拥有班级群管理权限。
- 任课老师默认是普通群成员，除非另行设置为班级管理员。
- 班级群成员可以查看本班成员名单，但不能查看手机号。

### 平台管理员隐身治理

- 管理员和总管理员通过后台权限管理所有班级群，不写入群成员表。
- 平台管理员不出现在成员列表、在线状态、入群记录或系统提示中。
- 平台管理员查看消息、冻结群、删除违规内容和处理申诉都写入不可由普通群管理员删除的审计日志。
- 平台管理员即使身份资料包含某个班级，也不以普通成员身份出现在该班级群。

### 普通群

- 所有正式登录且状态正常的非游客账号都可以创建普通群。
- 创建者成为群主，群主可以设置多个群管理员。
- 群主可以解散普通群；解散后保留审计记录，成员不可继续访问。
- 用户通过三种方式加入：指定邀请、搜索群号申请、扫描二维码申请。
- 指定邀请由被邀请人确认；群号和二维码申请由群主或群管理员审核。
- 普通群成员可以主动退出，群主退出前必须转让群主或解散群。
- 平台管理员主动创建或加入普通群时，以普通群成员身份出现；隐身管理只用于后台治理，不能用于匿名参与聊天。

## 群号、邀请与随机二维码

### 群号

- 每个群生成不可重复、便于输入的公开群号。
- 搜索结果只显示群头像、群名、群类型、成员数量和入群说明。
- 搜索结果不暴露完整成员名单、管理员姓名或班级身份详情。

### 邀请

- 群内具备邀请权限的用户可搜索有限校内公开资料并发出邀请。
- 邀请只能看到姓名、头像、学校、学院和公开身份，不返回手机号。
- 被邀请人确认后加入，拒绝或过期不建立成员关系。

### 随机二维码

- 二维码只包含随机邀请令牌和站内跳转地址，不包含群号或身份信息。
- 令牌使用密码学安全随机数生成，并保存摘要而非原始值。
- 默认有效期为 7 天，可设置一次性使用、提前失效或主动刷新。
- 二维码申请仍需群主或群管理员审核，不直接入群。
- 刷新二维码后旧令牌立即失效。

## 群聊页面与交互

### 桌面端

采用三栏协作布局：

- 左栏：班级群、普通群、未读数、入群申请和创建群聊。
- 中栏：群头部、消息历史、引用回复、快捷操作和消息输入区。
- 右栏：群资料、群号、二维码、公告、成员名单、权限与冻结状态。

中栏是主要滚动区域，群列表和成员栏各自独立滚动。消息输入区固定在中栏底部，消息历史上滑分页，页面本身不因消息增多无限增长。

### 手机端

采用“群列表 → 对话 → 群资料”三级页面：

- 群列表显示群类型、最后消息、时间和未读数。
- 对话页固定头部和输入区，返回时保持群列表滚动位置。
- 群资料作为独立页面或全屏抽屉，成员和管理操作按权限显示。

### 消息能力

首版支持：

- 文本、基础表情、图片表情、图片和文件。
- 回复引用、复制、本人消息撤回和管理员删除违规消息。
- 消息时间、发送状态、失败重试、未读数和已读位置。
- 群公告和系统消息。

不展示虚假的全员已读状态。只有已建立可靠游标的数据才显示实际已读信息。

## 实时通信与断线补偿

### 消息写入

1. 客户端为发送操作生成请求编号。
2. REST API 验证用户、群成员关系、群状态和内容限制。
3. 服务端以请求编号做幂等写入，生成群内递增消息序号。
4. 数据库提交成功后，服务端通知实时网关广播消息事件。
5. 客户端收到确认后把临时消息替换为正式消息。

### WebSocket

- 每个群使用独立实时频道。
- 连接凭证由业务 API 签发，包含用户、群和短期有效期。
- 实时网关只转发已持久化的消息事件，不直接决定成员权限。
- 客户端心跳、指数退避重连，并在页面隐藏时降低活动频率。

### 降级与补偿

- WebSocket 不可用时自动切换到可见页面 5 秒、后台页面 30 秒的增量轮询。
- 客户端保存每个群最后确认的消息序号。
- 重连或切换群时请求该序号之后的消息，去重后按序插入。
- 发送接口超时后可使用同一请求编号重试，不产生重复消息。

## 表情与图片内容

### 基础表情

使用 Unicode 标准表情，按常用、人物、手势、校园和物品等分类。基础表情不复制 QQ、微信的专有图形资源。

### 用户图片表情

- 用户可以上传、裁剪、命名、收藏和删除个人图片表情。
- 图片统一转换为 WebP，限制像素、体积、帧数和动画时长。
- 服务端验证真实 MIME 类型、扩展名和文件签名，随机化存储名。
- 图片表情默认仅本人可管理，在群内发送后按消息权限访问。

### 合规网络表情

- 网络搜索通过可配置来源适配器接入，仅展示明确允许转载、开放许可或授权来源。
- 保存来源链接、作者、许可类型和采集时间，不直接盗链。
- 用户确认后才加入个人表情库。
- 来源许可不明确时禁止收藏和发送，只允许打开原始页面。

### 举报与拦截

- 用户可举报图片表情和图片消息。
- 管理员可下架内容，文件摘要进入拦截列表，防止同一违规文件重复上传。
- 举报、处理结果和申诉均记录审计信息。

## 冻结与申诉

### 冻结状态

群被冻结后：

- 禁止发送新消息、邀请成员和审核入群申请。
- 保留历史消息、群资料和申诉入口。
- 成员可以查看明确的冻结原因和规则分类，不显示举报人身份。

### 申诉流程

1. 群主提交申诉说明和证明材料。
2. 系统限制同一冻结事件的重复申诉频率。
3. 平台管理员查看群资料、相关消息和历史治理记录。
4. 管理员决定恢复、维持冻结或永久关闭，并填写审核意见。
5. 结果通知群主和群管理员，完整过程写入审计日志。

## 数据模型

### 班级与分配

- `campus_classes`：班级 ID、学校、学院、班级名称、规范化班级键、班级群 ID、状态。
- `class_assignments`：班级 ID、用户 ID、班级职务、来源、有效状态和时间。

`campus_classes` 对“学校 + 学院 + 班级名称”建立唯一索引。`class_assignments` 对“班级 + 用户”建立唯一索引，一位老师可在多个班级存在记录。

### 群与成员

- `chat_groups`：群类型、群号、显示名称、头像、群主、班级关联、状态和创建时间。
- `chat_members`：群、用户、群角色、禁言状态、加入来源和最后已读序号。
- `chat_join_requests`：群号/二维码申请、申请人、审核状态和审核人。
- `chat_invites`：邀请人、被邀请人、有效期和确认状态。
- `chat_invite_tokens`：群、令牌摘要、有效期、最大使用次数和撤销状态。

班级群对班级 ID 建立唯一索引。平台管理员不写入 `chat_members`。

### 消息与治理

- `chat_messages`：群、群内序号、发送人、请求编号、消息类型、内容、引用消息和状态。
- `chat_read_cursors`：群、用户、最后已读序号和更新时间。
- `chat_appeals`：群、冻结事件、申诉内容、证明材料、审核状态和意见。
- `user_stickers`：用户、文件、来源、许可、文件摘要和状态。
- `sticker_favorites`：用户和图片表情收藏关系。
- `content_reports`：举报对象、原因、处理状态和结果。
- `chat_audit_logs`：操作者、动作、目标、详情和时间。

## API 边界

### 用户接口

- 获取我的群列表、群详情、成员和消息。
- 创建普通群、更新群资料、退出或解散普通群。
- 发送消息、撤回本人消息和更新已读游标。
- 搜索群号、申请入群、确认邀请和审核申请。
- 创建、刷新和撤销二维码邀请令牌。
- 上传、收藏、搜索和发送表情。
- 查看冻结原因并提交申诉。

### 班级管理接口

- 管理员查询班级、班级分配和同步状态。
- 设置学生班级职务。
- 分配老师到班级并设置职责。
- 手动重试班级群和成员同步。

### 平台治理接口

- 查询所有群和治理状态。
- 冻结、恢复或关闭群聊。
- 删除违规消息或下架图片表情。
- 审核冻结申诉。
- 查询管理员操作审计记录。

所有接口在服务端重新验证平台角色、班级职务、群角色和群状态。客户端是否显示按钮不构成授权依据。

## 性能与存储

- 身份库班级排序建立“学校、学院、班级、角色、姓名”组合索引。
- 群列表按用户和最近活动建立索引。
- 消息按“群 + 群内序号”建立唯一索引，历史消息使用游标分页，不使用大偏移量分页。
- 成员列表按群角色、班级职务和姓名建立稳定顺序。
- 群详情、成员数量和未读数使用短时缓存，成员变化后主动失效。
- 聊天页面只渲染可见窗口附近的消息，长历史使用虚拟列表。
- 图片和文件不存入 MySQL 二进制字段，只保存对象存储元数据和受控访问地址。

## 安全与隐私

- 群成员接口只返回业务所需的姓名、头像、公开身份和班级职务，不返回手机号、密码状态或内部管理字段。
- 用户只能查看自己的班级成员；普通群只能查看已加入群的成员。
- 搜索成员和群号均限流、限制结果数量并记录滥用信号。
- 文件上传限制格式、体积和访问权限，使用签名地址或受鉴权代理下载。
- WebSocket 凭证短期有效、限定群和用户，不能作为 REST 身份令牌使用。
- 实时网关与业务 API 之间使用独立内部签名密钥并防止重放。
- 平台管理员隐身访问和治理操作全部审计。
- 内容删除采用可审计状态变更，不直接抹除治理证据。

## 错误处理

- 班级资料不完整：不建群，身份库显示修复入口。
- 班级同步失败：事务回滚，记录可重试任务，不留下双班成员关系。
- 群号或二维码无效：统一返回“邀请已失效或不可用”，不泄露群状态。
- 群被冻结：发送接口返回冻结原因分类和申诉入口信息。
- WebSocket 失败：静默切换增量轮询并显示轻量连接状态，不阻断发送。
- 图片处理失败：不保存消息，向用户说明格式、大小或安全检查原因。
- 重复发送：按用户、群和请求编号返回原消息，不新增记录。

## 测试方案

### 数据与权限

- 相同学校、学院、班级且专业不同的学生进入同一班级群。
- 学校、学院或班级不同的同名记录不会串群。
- 学生换班后从旧群移除并加入新群，失败时事务回滚。
- 老师不因班级字段自动入群，明确分配后可加入多个班级。
- 班长和团支书可管理本班群，但不能访问平台管理接口。
- 管理员和总管理员能治理所有群，但不出现在成员和系统消息中。
- 普通用户不能读取其他班级完整成员名单。

### 群聊流程

- 普通群创建、群主转让、退出和解散流程正确。
- 指定邀请必须由被邀请人确认。
- 群号和二维码申请必须由群管理员审核。
- 二维码过期、撤销、刷新和一次性使用规则正确。
- 重复请求编号不会产生重复消息。
- WebSocket 断开后轮询补齐消息，重连后无丢失或重复。
- 冻结后禁止发送和加人，申诉审核后正确恢复或关闭。

### 表情和文件

- Unicode 基础表情无需文件存储即可发送。
- 非法 MIME、超限图片、危险文件和拦截摘要被拒绝。
- 合规网络表情保存来源和许可信息。
- 下架后的图片表情不能再次发送或重复上传。

### 前端与性能

- 身份库按班级稳定排序，跨页分组标识正确。
- 桌面三栏与手机三级页面无重叠、溢出和不可达操作。
- 群消息达到长历史规模时仍使用游标分页和虚拟列表。
- 页面隐藏后降低轮询频率，返回页面后补齐消息。
- 键盘可操作群列表、消息输入、表情面板和成员管理。

## 发布顺序与回退

1. 部署数据表和索引，保持现有功能不受影响。
2. 上线班级建模、身份库排序和后台班级职务管理。
3. 运行班级群同步预览，管理员确认后执行正式同步。
4. 上线 REST 群聊和轮询降级能力。
5. 上线 Cloudflare 实时网关，按配置逐步启用 WebSocket。
6. 上线图片表情、网络来源和冻结申诉能力。

每一步均使用功能开关。实时网关可独立关闭并回退到轮询；图片表情和网络来源可独立停用；班级群同步保留重试和审计记录，不通过删除数据回退。
