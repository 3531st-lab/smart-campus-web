# 登录身份联动设计

## 背景与目标

当前登录页按“学校、专业、学号/工号、手机号”录入，用户必须手动填写专业。身份库已经保存学校、专业、学号/工号和角色，因此可以在不改变现有登录核验规则的前提下，用学校与学号/工号自动匹配专业。

本次目标：

- 登录字段调整为“学校、学号/工号、专业、手机号”。
- 学校选项实时来自数据库中的有效身份记录。
- 选择学校并输入学号/工号后自动填充专业。
- 学生、老师登录分别按现有角色规则匹配。
- 数据库或联动接口暂时不可用时，仍保留手动登录回退路径。
- 不向未登录用户暴露姓名、手机号、班级、角色等额外身份信息。

## 不在本次范围

- 不改变短信验证码、密码登录、用户协议同意和游客模式流程。
- 不修改登录所需的学校、专业、学号/工号、手机号四项最终核验规则。
- 不在浏览器缓存完整身份记录或学校与专业的全量对应关系。
- 不改动学生身份库管理页的数据结构。

## 交互设计

### 字段顺序

登录表单按以下顺序展示：

1. 学校选择框。
2. 学号；老师模式下显示为工号。
3. 专业。
4. 手机号。
5. 验证码或密码。

### 学校选择

- 页面加载后，根据当前学生或老师模式请求数据库中存在且状态正常的学校列表。
- 学校按中文名称去重并排序。
- 若浏览器保存的上次学校仍在列表中，则优先选中；否则选择列表第一项。
- 切换学生/老师身份时重新加载对应学校列表，并清空已有学号/工号和专业匹配结果。
- 学校接口失败时，将学校控件切换为可手动输入状态，并显示轻量提示，避免数据库短时故障阻断登录。

### 专业自动填充

- 学号/工号输入达到最小长度后，停止输入 350 毫秒或离开输入框时触发查询。
- 查询条件为学校、学号/工号、登录身份类型。
- 新查询开始时取消上一条未完成请求，避免快速输入造成旧响应覆盖新结果。
- 匹配成功后填入专业，并将专业设为只读，显示“已从校园身份库匹配”。
- 学校、学号/工号或身份类型变化时立即清空旧专业结果。
- 未匹配时不返回任何个人资料，提示用户检查学校和学号/工号。
- 服务异常时允许用户手动填写专业，并提示“身份联动暂不可用，可继续手动填写”。

## 服务端设计

### 学校列表接口

`GET /api/auth/identity/schools?identityType=student|teacher`

返回：

```json
{
  "schools": ["泰州学院"]
}
```

规则：

- 仅统计状态为 `active` 的账号。
- `teacher` 只包含老师账号；`student` 与现有登录逻辑一致，包含非老师账号。
- 仅返回去重后的学校名称。
- 服务端缓存结果 5 分钟；身份库写入、导入或状态/角色变化后清除缓存。

### 专业匹配接口

`POST /api/auth/identity/major`

请求：

```json
{
  "school": "泰州学院",
  "studentNo": "240645136",
  "identityType": "student"
}
```

匹配成功返回：

```json
{
  "matched": true,
  "major": "数字经济"
}
```

规则：

- 学校、学号/工号和身份类型必须完整且经过长度限制与规范化。
- 仅匹配状态正常且身份类型一致的记录。
- 返回内容仅包含匹配状态和专业，不返回姓名、手机号、班级、学院、角色或内部 ID。
- 未匹配统一返回模糊错误，不区分学校不存在、账号不存在、状态异常或角色不符。
- 接口按 IP 与学校/账号组合限流，避免批量枚举。
- 查询日志不记录手机号，也不记录完整身份资料。

### 身份库查询

在 `student-store` 增加两个窄接口：

- `listActiveSchools(identityType)`：只读取去重学校名。
- `findMajorBySchoolAndAccount({ school, studentNo, identityType })`：只承担联动匹配，不替代最终登录校验。

内存数据模式与 MySQL 模式必须保持一致行为，以便本地开发和线上部署得到相同结果。

## 前端状态与可访问性

- 学校加载、专业查询、匹配成功、未匹配、服务异常分别有明确状态。
- 查询期间专业框显示加载状态，按钮与表单布局不发生跳动。
- 状态提示使用 `aria-live="polite"`，键盘用户可完整操作学校选择和登录表单。
- 自动填充不能抢走用户焦点。
- 手机端保持单列顺序；桌面端学号/工号与专业可并排，但 DOM 顺序仍为学号/工号在前。

## 安全与隐私

- 两个接口均为同源接口，复用现有请求防护和安全响应头。
- 专业查询设置独立的短周期速率限制，并限制字段长度。
- 不把专业映射或身份记录持久化到 `localStorage`；仅可记住学校名称。
- 前端显示统一失败文案，服务端不通过错误文本泄露账号状态或角色。
- 最终短信发送、验证码登录和密码登录仍调用现有四项身份核验，自动填充结果不能绕过认证。

## 测试方案

### 服务端测试

- 学生模式只返回存在有效非老师账号的学校。
- 老师模式只返回存在有效老师账号的学校。
- 学校列表去重、排序且不包含停用账号独有学校。
- 正确学校、账号和身份类型返回专业。
- 错误学校、错误账号、角色不符、停用账号均不返回专业。
- 响应不包含姓名、手机号、班级、学院、角色和内部 ID。
- 内存模式与 MySQL 查询参数和行为一致。

### 前端契约测试

- 登录页 DOM 顺序为学校、学号/工号、专业、手机号。
- 学校控件从身份学校接口加载。
- 学号/工号变化会触发专业查询并处理竞态取消。
- 身份切换会清空旧匹配结果。
- 接口异常时专业恢复为可编辑状态。

### 回归测试

- 学生短信登录、老师短信登录、密码登录和游客登录保持可用。
- 用户协议阅读与同意流程不变。
- 桌面端和手机端登录页无溢出、遮挡或字段错序。

## 发布与回退

- 先提交自动化测试，再实现身份库查询、API 和前端联动。
- 运行语法检查、完整测试套件和登录页桌面/手机视觉检查。
- 新接口失败时前端自动降级为手动输入，因此可独立回退服务端或前端改动。
- 发布后观察接口错误率、查询延迟和登录失败率，不记录用户输入的完整身份组合。
